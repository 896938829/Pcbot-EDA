# 文档规范

| 字段 | 值 |
| --- | --- |
| 定位 | 本仓库文档写作与组织的最小共同约定 |
| 读者 | 所有文档贡献者（AI Agent 与人类） |
| 状态 | draft |
| 最后更新 | 2026-04-16 |

---

## 1. 语言

- 叙述正文：**中文**。
- 专有名词、API 名、命令行：**保留英文原样**（Godot、SQLite、JSON-RPC、GDScript、`.sch.json` 等）。
- 中英文混排时，在中文与英文 / 数字之间空一格（如 `Godot 4.6` 不是 `Godot4.6`）。
- 避免口语化（"咱们"、"搞"、"就行了"）；可以简洁但不随意。

## 2. 编码与行尾

- 编码：UTF-8（与 `.editorconfig` 一致）。
- 行尾：LF（与 `.gitattributes` 一致）。
- 文件末尾：保留一个空行。

## 3. 位置与命名

| 内容类型 | 位置 |
| --- | --- |
| 仓库入口、路线图、速览 | `docs/工程文档.md` |
| 架构设计与 ADR | `docs/architecture/` |
| 里程碑 / 功能具体规划 | `docs/plan/` |
| 具体特性 / 调研规格 | `docs/specs/<YYYY-MM-DD>-<主题>.md` |
| 代码 / 文档约定 | `docs/conventions/` |
| Skills YAML | `docs/skills/<namespace>/<method>.yaml` |
| 图片与附件 | `docs/assets/<主题>/` |
| 输入资料（需求、技术栈） | 直接放 `docs/` 下 |

- 文件名用中文时，**不含空格**，分词用"-"或直接相连（示例：`架构设计.md`、`实现计划-M1.md`）。
- ASCII 文件名一律 `kebab-case.md`。
- ADR：`ADR-<三位数编号>-<主题>.md`，例如 `ADR-001-headless-cli.md`。

## 4. 标题层级

- 每份文档只有一个 H1（文件顶部）。
- H2 为主要章节，编号建议形如 `## 3. 标题`，便于跨文件引用。
- H3 按需；**禁止**跳级（H2 下直接 H4）。
- 不要在标题里塞代码或链接；要强调的术语放正文第一段。

## 5. 结构模板

每份 `.md` 文件顶部紧跟 H1 之后放一张元数据表：

```markdown
# 标题

| 字段 | 值 |
| --- | --- |
| 定位 | 一句话说清楚这篇文档存在的理由 |
| 读者 | 谁该读这篇 |
| 状态 | draft / stable / deprecated |
| 最后更新 | YYYY-MM-DD |

---
```

常用章节骨架（可裁剪）：

1. **背景 / Context** — 为什么写这个。
2. **决策 / 方案** — 是什么。
3. **权衡 / 备选** — 否决了哪些，为什么。
4. **影响 / 后果** — 谁被影响、要做什么迁移。
5. **验证 / 验收** — 怎么知道它对。
6. **参考** — 相关文档、外链。

### 5.1 ADR 模板

```markdown
# ADR-001: 标题

| 字段 | 值 |
| --- | --- |
| 状态 | proposed / accepted / superseded |
| 日期 | YYYY-MM-DD |
| 相关 | 链接到其他 ADR / 文档 |

## Context
……

## Decision
……

## Consequences
……

## Alternatives Considered
……
```

## 6. 图表

- 优先 **ASCII 图**或 **Mermaid**——纯文本、git diff 可见、AI 可读。
- 需要真实图片时：放 `docs/assets/<主题>/`，使用 PNG/SVG，文件名 `kebab-case`。
- 图要有编号与简短说明（如 `图 1：CLI 请求流程`）。
- 不要嵌入外链图片（链接失效 = 死图）。

## 7. 链接与引用

- **文档间用相对路径**：
  - 同目录：`[xx](xx.md)`
  - 子目录：`[xx](plan/实现计划-M1.md)`
  - 向上：`[xx](../架构设计.md)`
- **锚点**：小写、用 `-` 连接（如 `#5-结构模板`）。中文章节锚点建议给出中英双写，或直接引用章节编号。
- **外链**：尽量指向规范或官方文档；临时链接（issue、PR）放参考区，不散嵌正文。
- 改名 / 移动文件时 `git grep` 搜一遍引用。

## 8. Skills YAML 格式

位置：`docs/skills/<namespace>/<method>.yaml`。模板：

```yaml
name: schematic.place_component
description: 在原理图页面上放置一个元件实例
triggers:
  - "AI 已选定元件且需要把它摆到原理图上"
  - "正在构建初始电路拓扑"
parameters:
  page_id:
    type: string
    required: true
    description: 目标页面 id
  component_ref:
    type: string
    required: true
    description: 元件库引用（如 library://vendor/part_number）
  pos_nm:
    type: "[int, int]"
    required: true
    description: 放置位置，纳米整数坐标
  rotation_deg:
    type: int
    required: false
    default: 0
    description: 顺时针角度，仅允许 0/90/180/270
examples:
  - description: 放置一个 NE555
    request: |
      {"jsonrpc":"2.0","id":1,"method":"schematic.place_component",
       "params":{"page_id":"p1","component_ref":"library://ti/ne555","pos_nm":[10000000,10000000]}}
    response: |
      {"jsonrpc":"2.0","id":1,"result":{"uid":"c_0001"}}
common_errors:
  - code: USER_ERROR
    when: "component_ref 不存在于已加载的元件库"
    remedy: "先调用 library.add_component 或 library.search 确认 ref"
  - code: USER_ERROR
    when: "rotation_deg 不是 0/90/180/270 之一"
    remedy: "规范化到四个正交角度"
see_also:
  - schematic.connect
  - library.search
```

规则：

- `name` 与 CLI 命令完全一致（`<namespace>.<method>`）。
- `triggers` 用短句、描述情境；给 AI 判断"现在该不该用这个命令"。
- `examples` 至少 1 个完整 request/response。
- `common_errors` 至少 2 条，覆盖最易踩的两种。

## 9. 代码与命令片段

- 用围栏代码块，标注语言：```` ```gdscript ````、```` ```bash ````、```` ```json ````。
- 可直接复制执行；若平台差异，用注释分别标出：
  ```bash
  # Unix
  godot --headless --path . -s cli/main.gd -- '{...}'

  # Windows PowerShell（注意引号转义）
  godot --headless --path . -s cli/main.gd -- '{\"jsonrpc\":\"2.0\",...}'
  ```
- 长命令可折行；折行用 `\`（Unix）或 `` ` ``（PowerShell），并在文档里说明适用 shell。

## 10. AI 可读性

- 段落短，每段一个要点；避免"此"、"该"、"它"等无明确指代的代词。
- 关键决策放小标题下的第一段第一句。
- 有顺序的内容用有序列表；并列用无序列表。
- 表格优先于长段描述；任何"A 或 B 哪种"的讨论都该能压缩成 2 列表格。
- 避免"详见后文"的悬挂引用；要么给锚点，要么不写。

## 11. 状态流转

| 状态 | 含义 |
| --- | --- |
| `draft` | 写作中 / 审核中，内容可能大改 |
| `stable` | 决策落地，引用时可信 |
| `deprecated` | 仍保留但已不是事实；必须在顶部指明接替者 |

- 文档不删除只 deprecate；理由：git 历史 + 外部链接都可能在引用旧文件。
- 例：
  ```markdown
  | 状态 | deprecated → 参见 [新文档](../new-doc.md) |
  ```

## 12. Review 要点

提交文档变更前自检：

- [ ] 元数据表齐全且日期已更新。
- [ ] 无死链（相对路径存在；锚点对应章节存在）。
- [ ] 无 CRLF 行尾（`git diff --check`）。
- [ ] 无机器相关绝对路径。
- [ ] 新增术语在 `docs/工程文档.md` 的词汇表或本文附录中定义。
- [ ] 与代码实现一致（若文档先行，用 `状态: draft` 标明）。
