## 1. 为什么写这份

需求文档说"AI 为设计主体"、架构文档给出 CLI/文件格式、规范文档约束产物形式。**三者没有回答的问题**：

- AI 每次调用进场时，先读什么、后读什么？
- AI 如何判断自己"做对了"，不靠人类告诉？
- AI 遇到违例/失败时的分岔决策怎么走？
- 什么时候停下来，把控制权交还人类？

本文件回答以上。是 **行为层哲学**，不是代码层规范。

## 2. 核心假设

- AI 是**无状态短生命周期 actor**：每次 CLI 调用 = 一次新启的进程+一次新鲜的 AI 会话。跨调用记忆**必须**在文件里。
- 编辑器是**被动响应器**：不替 AI 思考、不缓存 AI 上下文、不保存对话历史。
- **源文件 = 状态真相** / **git = 历史真相** / **`.pcbot/` = 上次调用遗嘱**。三者以外皆为衍生，不得信赖。
- AI 的"增效"度量 = **无效调用占比 ↓** + **往返轮次 ↓** + **人类修正事件数 ↓**。不是"AI 能跑多少命令"。

## 3. AI 的单次调用心流

```
 ┌─────────────────────────────────────────────────────────────────┐
 │  感知 (Sense)                                                    │
 │   1. 读 .pcbot/last-run.json  ← 上次做了什么、错在哪              │
 │   2. 读 .pcbot/diagnostics.jsonl (tail) ← 未解规则违例            │
 │   3. 按需读源文件（不盲读全部）                                   │
 │   4. skills.get 要调用命令的说明                                  │
 └────────────────────┬────────────────────────────────────────────┘
                      ▼
 ┌─────────────────────────────────────────────────────────────────┐
 │  规划 (Plan)                                                     │
 │   - 本次只做一件事；多步拆成多次 CLI 调用                          │
 │   - 若上次失败，优先分析失败原因再前进                             │
 │   - 若违例未解，优先消除违例再扩张                                 │
 └────────────────────┬────────────────────────────────────────────┘
                      ▼
 ┌─────────────────────────────────────────────────────────────────┐
 │  执行 (Act)                                                      │
 │   - 单条 JSON-RPC 调用                                            │
 │   - stdout JSON-RPC 响应 / stderr 诊断 / exit_code                │
 └────────────────────┬────────────────────────────────────────────┘
                      ▼
 ┌─────────────────────────────────────────────────────────────────┐
 │  反馈 (Reflect)                                                  │
 │   - 读 .pcbot/last-run.json 验证 touched_files 与预期一致          │
 │   - 若 exit_code != 0，走"失败分岔"                               │
 │   - 若成功且改了源文件，读 .pcbot/commit-msg 调 git commit         │
 └─────────────────────────────────────────────────────────────────┘
```

**设计要点**：感知 → 反馈 闭环发生在**一次 AI 决策内**，CLI 调用本身是原子的一拍。AI 的"心流"是多次 AI 决策的串联，每拍之间状态全部落地文件。

## 4. 读取优先级（进场顺序）

AI 进入一个已存在的工程时，按如下顺序读文件。**先读高层，再读局部**；不必每项都读满。

| 优先级 | 文件 | 读的目的 | 止读条件 |
| --- | --- | --- | --- |
| P0 | `.pcbot/last-run.json` | 了解上次失败/成功 | 恒读 |
| P0 | `docs/工程文档.md` 的 §2 哲学 + §5 路线图 | 知道当前里程碑能做什么 | 会话首次 |
| P1 | `.pcbot/diagnostics.jsonl`（tail 10 条） | 未解违例清单 | 恒读 |
| P1 | `*.pcbproj` | 工程清单、引用的库与页面 | 恒读 |
| P2 | `skills.list` → `skills.get <method>` | 命令用法 | 仅对本次要调的命令 |
| P3 | 具体 `*.sch.json` / `*.sym.json` | 当前要改的文件 | 仅打开要改的文件 |
| P4 | `docs/architecture/架构设计.md` §4–§7 | 遇到数据模型或协议疑问时 | 按需 |
| P5 | `docs/conventions/代码规范.md` §8 | 写文件前确认格式 | 按需 |
| P6 | `git log -- <file>` | 搞清楚历史变更原因 | 按需 |

**禁止**：
- 盲读 `Runtime/` 下所有脚本来"了解系统"。实现细节与 AI 任务正交。
- 把 `docs/` 全部 dump 进上下文。浪费 token，且多数不需要。
- 重复读同一文件在同一次决策内。

## 5. 执行纪律

### 5.1 单次调用一件事

- 一个 JSON-RPC 请求只触发一条命令。不要拼多命令序列。
- 批量意图（"放 10 个元件"）拆成 10 次独立调用。每次都独立产 `last-run.json`，便于局部失败回退。

### 5.2 失败分岔

读 `last-run.json.exit_code`：

| exit_code | 含义 | 该做的 |
| --- | --- | --- |
| `0` | 成功 | 验证 `touched_files`；若要连续做事，开新调用 |
| `1` | 用户错（参数非法） | **不要重试原命令**。读 `errors[]`，修正参数；若 params 正确怀疑 skills 过期，`skills.get` 重查 |
| `2` | 系统错（I/O、引擎） | 最多重试 1 次；连续 2 次 `2` 即**升级人类**（见 §7） |
| `3` | 违反设计规则 | 读 `diagnostics.jsonl` 最新行，针对 `rule_id` 做消除动作；**不要**用 `--force` 之类参数绕过 |

### 5.3 写操作后必提交

- 成功的写命令会生成 `.pcbot/commit-msg`。
- AI 下一步 = `git commit -F .pcbot/commit-msg`（可补充 body 解释"为什么"，不改 subject）。
- **禁止**多命令累积后一次性 commit。一个命令 = 一个 commit 是原子性保证，也是人类审核的颗粒度。
- commit subject 已由编辑器标准化；AI 改写自由度受限，有意为之。

### 5.4 不猜

- 不知道命令签名 → `skills.get`。
- 不知道元件是否在库 → `library.search`。
- 不知道网络连到哪 → 读 `*.sch.json`。
- 不知道上次做了什么 → `run.last` 或直接 `cat .pcbot/last-run.json`。
- 全部手段用尽仍不确定 → 停下来问人类（见 §7）。

**禁止猜到"大概对"然后写文件**。错误写入的 diff 比没写还糟——人类要 review 一次噪声。

## 6. 设计过程的颗粒度

一个"设计任务"如何分解：

```
Task: 做一个 LED 闪烁电路
└── 子任务 1: 建元件符号（若库里没有）
│   ├── CLI: symbol.create NE555
│   ├── CLI: symbol.edit_pin × 8
│   ├── git commit
│   └── CLI: symbol.create LED / 电阻 / ...
├── 子任务 2: 建库条目
│   └── CLI: library.add_component × N + git commit × N
├── 子任务 3: 建原理图页面
│   ├── CLI: project.new / schematic.new
│   └── git commit
├── 子任务 4: 放元件
│   └── CLI: schematic.place_component × N + git commit × N
├── 子任务 5: 连线
│   └── CLI: schematic.connect × N + git commit × N
├── 子任务 6: 自检
│   ├── CLI: check.basic
│   ├── 若有违例 → 回到子任务 4/5 消除
│   └── git commit
└── 完成
```

**颗粒度原则**：
- **叶节点 = 一次 CLI 调用**。
- **中间节点 = 可用一句话向人类描述的设计意图**。
- 每个叶节点后都有 git commit；每个中间节点完成后都有 `check.basic`。
- 违例出现时，**就地回到最近的叶节点**修正，不要"之后一起解决"。

## 7. 升级人类的触发条件

AI 必须把控制权交还人类的情形：

1. **需求歧义**：设计目标人类给的是"做个闪烁电路"，但频率/占空比/电压未定，影响选件。**不得**擅自代选。
2. **规则违例无法消除**：连续 3 次尝试都产出同 `rule_id` 违例。
3. **连续 2 次 exit_code == 2**：系统错误层面，AI 无法判别根因。
4. **跨 session 冲突**：AI 预期的状态与 `last-run.json` / 源文件不一致（可能人类手动改过）。
5. **架构级决策**：需要改 `docs/architecture/` 或新增 ADR 才能继续。
6. **破坏性操作**：`git reset --hard`、`rm` 设计文件、跨页面大范围重构——必须人类确认。

升级方式：AI 在自己的对话中**停下**并说明原因，不再发 CLI 调用。**不要**尝试在编辑器里塞"求助"命令——编辑器是被动的。

## 8. 与人类审核者的协作切面

| 人类审核时机 | AI 该做的准备 |
| --- | --- |
| 每次 git commit 后 | commit-msg 含命令/参数摘要；`touched_files` 明确 |
| `check.basic` 出违例时 | `diagnostics.jsonl` 的违例条目附 `msg` 描述 |
| 一个子任务完成时 | 追加一条 `docs/specs/<date>-<topic>.md` 迷你规格（可选） |
| 里程碑完成时 | `git log --oneline` 可读、每条 commit 可独立回退 |

**AI 不代写 PR 描述 / Release Notes**：那是人类叙事，AI 产出只应填事实区（commit 列表、违例清单）。

## 9. "AI 友好度"可度量指标

给编辑器开发者的反向约束——功能新增前，想想是否会让这些指标恶化：

| 指标 | 含义 | 目标 |
| --- | --- | --- |
| **首命令可执行率** | AI 根据 skills 直接调成功的比例 | ≥ 95% |
| **平均命令往返** | 完成一个叶任务需要的 CLI 调用数 | ≤ 1.2（含 1 次 skills.get 的 overhead） |
| **失败消解深度** | 失败后平均需要几次调用才恢复 | ≤ 2 |
| **误读率** | AI 因文档与实现不一致而错调的比例 | ≤ 1% |
| **升级人类率** | 每 100 次调用触发 §7 的次数 | ≤ 3 |

**若某新功能让任一指标恶化 > 20%，重新评估这个功能是否该进 CLI**。

## 10. 反模式（常见浪费）

| 反模式 | 症状 | 原因 | 纠正 |
| --- | --- | --- | --- |
| **读满仓库** | AI 开局 `find . -name "*.gd"` 然后全部读 | 把实现细节当需要的上下文 | 按 §4 优先级读 |
| **盲打命令** | 不查 skills 直接猜参数 | 以为省一次调用 | `skills.get` 调用 ≪ 失败重试成本 |
| **堆积式 commit** | 10 个命令后一次性提交 | 以为省事 | 每命令一次 commit（强约束） |
| **沉默失败** | `exit_code != 0` 后不读 last-run 接着发下一条 | 没有反馈环 | §5.2 分岔必走 |
| **重复违例** | 同 `rule_id` 三次写不同文件 | 不读 `diagnostics.jsonl` | 每次进场先 tail 诊断 |
| **绕过规则** | 找 `--force` / 直接改源文件 JSON | 不信 CLI | 规则是人类需求；绕规则 = 绕人 |
| **幻觉重放** | 想从 `git log` 重建状态 | 把历史当状态 | 状态在源文件，历史只是审计 |
| **多余中间文件** | 自建 `my-notes.md` / `ai-state.json` | 不信任 `.pcbot/` | §11 "可增减" |

## 11. 何时可增加中间文件

永远从"能不能不加"开始问。必须新增时，走以下流程：

1. 证明：git / 源文件 / `.pcbot/` 三处均**无法**承载该信息。
2. 走 ADR（`docs/architecture/ADR-<n>-<主题>.md`）。
3. 更新 [架构设计 §4.2](architecture/架构设计.md#42-ai-中间文件务实最小集) 与 [代码规范 §8.2](conventions/代码规范.md#82-中间文件pcbot服务-ai)。
4. 更新本文件 §10 的反模式表（若有可能被滥用）。

**不**满足以上者，禁止新增。

## 12. 对 Skills YAML 的特殊态度

- Skills 是 AI 与编辑器之间的**契约**。contract 变，行为必变。
- Skills 更新时 AI 的旧知识瞬间过期——因此每次调用陌生命令前 `skills.get` 是便宜保险。
- 编辑器开发者在改 Skills 前先跑 [实现计划-M1](plan/实现计划-M1.md) P9 的 e2e，确认不破坏已验证路径。
