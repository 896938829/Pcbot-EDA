## 1. 架构目标与非目标

### 1.1 目标

- **AI 可驱动全流程**：元件创建 → 原理图 → 布局 → 布线 → Review → Debug，任一环节都能仅通过 CLI 完成。
- **CLI 与 GUI 同底**：两者调用的是同一套领域模型与服务；GUI 只是额外加载 UI 层。
- **Git 友好**：所有设计文件在 `git diff` 下可读、可评审、可合并。
- **精度与可复现**：整数 nm 坐标 + 稳定序列化，保证同输入同输出。
- **可扩展**：模块化插件架构，新增功能（如新文件格式、新 DRC 规则）不需要改核心。

### 1.2 非目标

- 不追求与 KiCad / Altium 的文件格式完全兼容（互操作转换器放 M4+）。
- **编辑器不内置 AI 推理**：无 LLM 调用、无本地模型。AI 在进程外。
- 不做可视化决策树的 UI（决策透明度交给 AI Agent 侧）。
- 不提供历史版本管理 UI：依赖 git，编辑器仅有本次运行时的 undo/redo。

## 2. 系统分层

```
┌─────────────────────────────────────────────────────────────────────┐
│  入口层                                                              │
│   cli/              命令路由 · JSON-RPC 解析 · 返回码                 │
│   Scenes/Main.tscn  GUI 入口（headless 模式不加载）                   │
├─────────────────────────────────────────────────────────────────────┤
│  UI 层（仅 GUI 模式）                                                │
│   Runtime/ui/        Control 节点 · 停靠面板 · 视口                  │
├─────────────────────────────────────────────────────────────────────┤
│  领域模块                                                            │
│   Runtime/modules/symbol/      符号编辑服务                           │
│   Runtime/modules/library/     元件库服务                             │
│   Runtime/modules/schematic/   原理图服务                             │
│   Runtime/modules/pcb/         PCB 服务（M2+）                        │
├─────────────────────────────────────────────────────────────────────┤
│  核心层                                                              │
│   Runtime/core/unit_system.gd   nm ↔ mm/mil 单位换算                  │
│   Runtime/core/event_bus.gd     域事件总线                            │
│   Runtime/core/logger.gd        结构化日志                            │
│   Runtime/core/error.gd         错误码与结构化错误                    │
├─────────────────────────────────────────────────────────────────────┤
│  I/O 层                                                              │
│   Runtime/io/json_stable.gd     稳定 JSON 序列化（键序、LF）          │
│   Runtime/io/yaml.gd            YAML 读写                             │
│   Runtime/io/jsonl.gd           JSONL 追加写                          │
│   Runtime/io/svg.gd             SVG 读写（符号）                      │
│   Runtime/io/sqlite.gd          SQLite 封装（元件库索引）             │
│   Runtime/io/project_fs.gd      设计工程目录抽象                      │
├─────────────────────────────────────────────────────────────────────┤
│  引擎与平台                                                          │
│   Godot 4.6 (GL Compatibility, Jolt Physics) · GDScript · C++ 扩展    │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.1 依赖规则

- 上层可依赖下层，下层不可依赖上层。
- `modules/<domain>/` 之间禁止直接引用对方内部脚本，只能通过 `core/event_bus` 发事件，或在入口（`cli/main.gd`、UI 加载点）显式构造并按需注入公开服务来通信。M1/M1.1 不引入 DI 容器，详见 [ADR-0004](ADR-0004-no-di-container-in-m1.md)。
- I/O 层只做"字节 ↔ 数据结构"的转换，不嵌业务规则。

## 3. 进程模型

**默认：单进程 headless。** AI 每次调用一次性启动 Godot，执行一条 JSON-RPC 命令后退出。状态完全由设计工程目录（中间文件）维持。

```
AI Agent                            Pcbot headless
────────                            ──────────────
   │                                     │
   │── spawn: godot --headless ──────▶ load 引擎
   │── stdin: JSON-RPC request ──────▶ cli/main.gd 解析
   │                                     │ 调用领域服务
   │                                     │ 读/写 设计文件
   │                                     │ 覆盖写 .pcbot/last-run.json
   │◀──── stdout: JSON-RPC response ─────│
   │◀──── stderr: 日志 ──────────────────│
   │                                     │
   │                                 exit(0..3)
```

**GUI 模式**：同一份项目代码，双击 Godot 启动时加载 `Runtime/ui/` 与 `Scenes/Main.tscn`，其余模块完全复用。

**为什么不选择长连接 daemon**（M1 范围）：
- headless 启动成本（目标 ≤ 3s）已可接受。
- 无状态进程消除了"AI 端崩溃后 daemon 脏状态"的复杂性。
- 文件系统 SSOT 保证任何观察者（AI、GUI、第三方工具）看到的都是同一份真相。
- 如后续证明启动开销不可接受，再把 daemon 作为可选模式加回来。

## 4. 数据模型

全部为 Godot `Resource` 子类（便于序列化、引用、测试）。

### 4.1 核心实体

| 类型 | 文件形态 | 关键字段 |
| --- | --- | --- |
| `ComponentSymbol` | `*.sym.json` + 配套 `*.sym.svg` | `id`、`name`、`pins[]`、`graphic_svg_ref`、`metadata{}` |
| `ComponentFootprint` | `*.fp.json`（M2 用） | `id`、`pads[]`、`outline`、`3d_model_ref?` |
| `Component`（库条目） | 元件库 SQLite 索引 + JSON 源 | `id`、`manufacturer`、`part_number`、`symbol_ref`、`footprint_refs[]`、`parameters{}` |
| `Schematic` | `*.sch.json` | `pages[]`、`placements[]`（引用 Component）、`nets[]`、`wires[]` |
| `DesignProject` | `*.pcbproj` | `schematic_refs[]`、`library_refs[]`、`pcb_ref?`、`settings{}` |

### 4.2 AI 中间文件（务实最小集）

**设计原则**：只写**别处没有**的信息。历史走 git，状态走源文件，本节文件**只服务 AI 的下一步调试与决策**。

| 类型 | 文件形态 | 用途 |
| --- | --- | --- |
| `LastRun` | `.pcbot/last-run.json`（每次 CLI 调用覆盖写） | AI 下次调用前 `cat` 一条命令读到上次运行的错误、告警、触达文件 |
| `Diagnostics` | `.pcbot/diagnostics.jsonl`（仅 `check.*` / DRC / ERC 命令追加） | 累计未解决的规则违例与告警；可在源文件提交后打标"已消除" |
| `CommitMsg` | `.pcbot/commit-msg`（成功变更时生成） | 结构化 commit 信息模板（含命令名、参数摘要、触达文件），AI 自行 `git commit -F` |

**为什么不再有 `*.ops.jsonl` 与 `*.snap.yaml`**：

- **ops 与 git 重叠**：git commit 已完整记录"谁、何时、改了什么 diff"；命令名/参数塞入 commit body 即可，额外的 ops 文件 = 两份真相 + 维护成本。
- **snap 与源文件重叠**：`*.sch.json` 等稳定序列化后本身就是状态真相，再维护一份 YAML 投影必然漂移。
- **hash + 可重放**是推测性功能：M1 没有环境锁（元件库版本、规则版本），重放不可靠；不为投机功能建机制。M3+ 视情况再评估。

**`.pcbot/last-run.json` 记录结构**：

```json
{
  "ts": "2026-04-17T10:00:00Z",
  "command": "schematic.connect",
  "params": {"page_id": "p1", "net": "N17", "pins": ["U1.3", "R1.2"]},
  "exit_code": 3,
  "errors": [
    {"code": "RULE_VIOLATION", "rule": "net.floating", "ref": "N17", "msg": "网络未接入任何引脚"}
  ],
  "warnings": [],
  "touched_files": ["demo/led_blink.sch.json"]
}
```

**`.pcbot/diagnostics.jsonl` 记录结构**：每行严格 JSON。

```json
{"ts":"2026-04-17T10:00:00Z","rule_id":"net.floating","severity":"error","ref":"N17","msg":"...","resolved_by_commit":null}
```

- `resolved_by_commit`：后续命令若消除该违例，追加一条新记录把 `resolved_by_commit` 填为当前 HEAD sha；不改已有行（JSONL 追加语义）。
- 该文件**入 git**（属项目质量史）。

**`.pcbot/` 目录本身**：

- `last-run.json` 与 `commit-msg` 为瞬态，**入 `.gitignore`**。
- `diagnostics.jsonl` 入 git。
- 目录由 CLI 首次运行自动创建。

## 5. 文件格式

所有文本文件：**UTF-8 + LF + 末尾换行**。

| 扩展 | 格式 | 稳定化要点 |
| --- | --- | --- |
| `.pcbproj` | JSON | 键字典序、2 空格缩进、相对路径引用 |
| `.sch.json` | JSON | 同上；几何坐标整数 nm；网络按 `id` 排序 |
| `.sym.json` | JSON | 同上；引脚按编号排序 |
| `.sym.svg` | SVG | 单位 mm，viewBox 固定，无 Inkscape 私有属性 |
| `.fp.json` | JSON | 同上；焊盘按 `number` 排序 |
| `.pcbot/last-run.json` | JSON | 每次 CLI 调用覆盖写；**不入 git** |
| `.pcbot/diagnostics.jsonl` | JSONL | 每行严格 JSON，追加写，时间戳 ISO8601 UTC；**入 git** |
| `.pcbot/commit-msg` | 文本 | 结构化 commit 信息模板；**不入 git** |
| `.pcbotlib.sqlite` | SQLite | **不入 git**；可从 JSON 源重建 |

## 6. 坐标与单位系统

- **内部单位**：`int64` 纳米（nm）。1 mm = 1,000,000 nm。
- **展示单位**：mm（默认）或 mil（工艺图层常用）；UI 层与人类可读导出在展示单位上。
- **API 边界**：所有跨模块接口用 nm 整数；禁止在公共 API 暴露 `float` 坐标。
- **Godot world 映射**：渲染时除以固定缩放常量（如 `10_000` nm/world_unit）再交 Godot 渲染器，确保浮点误差只在"视觉"而非"数据"上。
- **边界值**：`int64` 最大 ~9.2e18 nm，远超任意 PCB 尺寸。

## 7. CLI 协议

### 7.1 传输

- 请求：stdin 读取一行 JSON-RPC 2.0 请求对象（或请求数组）。
- 响应：stdout 写一行 JSON-RPC 2.0 响应对象（或数组）。
- 诊断：stderr 输出结构化日志行 `<LEVEL> <timestamp> <module>: <message>`。
- 换行：一律 `\n`。

### 7.2 退出码

| 码 | 含义 |
| --- | --- |
| `0` | 成功 |
| `1` | 用户错（参数非法、引用不存在） |
| `2` | 系统错（I/O 失败、引擎异常） |
| `3` | 违反设计规则（DRC/ERC，M2+） |

### 7.3 M1 首批命令清单

| 命名空间 | 方法 | 用途 |
| --- | --- | --- |
| `project` | `new` / `open` / `info` | 创建/打开/查询工程 |
| `library` | `list` / `add_symbol` / `add_component` / `search` | 元件库操作 |
| `symbol` | `create` / `edit_pin` / `export_svg` | 符号编辑 |
| `schematic` | `new` / `add_page` / `place_component` / `connect` / `annotate` | 原理图编辑 |
| `skills` | `list` / `get` | 列出/获取 Skills YAML |
| `check` | `basic` | 基础自检（无网络、未放置元件等）；产出 `.pcbot/diagnostics.jsonl` 追加记录 |
| `run` | `last` | 读取 `.pcbot/last-run.json`（若 AI 会话中断后想恢复上次上下文） |

M2+ 追加：`pcb.*`、`route.*`、`drc.*`、`erc.*`、`export.*`。

## 8. Skills 引导系统

- 位置：`docs/skills/<namespace>/<method>.yaml`。
- 作用：告诉 AI Agent「在什么情境下用这个命令 / 典型参数 / 常见错误 / 示例调用」。
- AI 通过 `skills.list` 列出全部、`skills.get {name}` 取单条。
- 格式：见 [文档规范 §8](../conventions/文档规范.md#8-skills-yaml-格式)。

## 9. 事件总线

`Runtime/core/event_bus.gd` 单例。

- **域事件**：`component.added`、`schematic.net.changed`、`rule.violated`、`run.completed`。
- **订阅**：模块在 `_ready()` 中注册；测试用例直接调 `emit_signal` 验证。
- **原则**：事件只传数据快照（不可变 dict），订阅者不得反向回调发布者内部状态。

## 10. 性能红线（M1 目标）

| 场景 | 指标 |
| --- | --- |
| 原理图 1k 元件 / 5k 网络 下单次编辑操作 | < 50 ms |
| CLI 冷启动到返回第一条响应 | ≤ 3 s |
| `*.sch.json` 读写 round-trip（1k 元件） | < 200 ms |
| 元件库搜索（10k 条） | < 100 ms |

M2+ 指标另行设定（布线、DRC 并发扫描）。

## 11. 扩展点

| 扩展点 | 机制 |
| --- | --- |
| 新 CLI 命令 | `Runtime/modules/<domain>/commands.gd` 调 `CommandRegistry.register()` |

## 12. 技术栈决策速览

| 领域 | 选择 | 一句话理由 |
| --- | --- | --- |
| 引擎 | Godot 4.6 | 开源 MIT、2D/3D 兼顾、节点树贴合 EDA 层次结构 |
| 主语言 | GDScript | 引擎原生、热重载、写 UI/业务最顺手 |
| 热点语言 | C++（GDExtension）/ C# | 空间索引、DRC 扫描需要接近原生性能 |
| 物理 | Jolt | 3D 预览（M4）需要稳定快速的 3D 物理 |
| 渲染 | GL Compatibility | 覆盖最广的显卡基线，EDA 用户机器差异大 |
| 数据库 | SQLite | 本地零配置、单文件、成熟可靠 |
| 设计文件 | JSON / YAML | git diff 友好、人类可读、工具链成熟 |
| 诊断日志 | JSONL | 追加写原子、逐行可 grep；`.pcbot/diagnostics.jsonl` 专用 |
| 符号图形 | SVG | 矢量、人类可读、第三方工具生态完整 |
| 版本控制 | Git + LFS | 事实标准；大模型/贴图走 LFS |
| 构建 | SCons | Godot 原生，C++ 扩展与引擎自编译都自然 |
| 测试 | GUT | Godot 生态内的主流单元测试框架 |
| CLI 协议 | JSON-RPC 2.0 | 规范清晰、错误码标准、AI 侧无歧义 |

## 13. 风险与缓解

| 风险 | 后果 | 缓解 |
| --- | --- | --- |
| 大规模原理图 / PCB 渲染卡顿 | 10k+ 元件时编辑器冻结 | 视口裁剪 + 四叉树 + 脏矩形重绘；必要时 `MultiMeshInstance2D` |
| GDScript `int64` 运算性能 | 热路径（DRC 扫描）拖慢 | M2 起把空间索引与 DRC 下沉到 C++ GDExtension |
| headless 启动开销大 | AI 每次调用 3s+ 变成 10s+ | 保留 daemon 模式做备选；缓存 `.godot/` 导入产物；按需懒加载模块 |
| JSON 序列化不稳定 | git diff 噪声 | 统一走 `Runtime/io/json_stable.gd`；CI 加 round-trip 测试（M2+） |
| 中间文件蔓延 | AI 被冗余文件误导、维护两份真相 | 严守"别处没有才写"原则；`last-run.json` / `diagnostics.jsonl` 之外不新增中间文件，需新增走 ADR |
| Godot `Resource` 二进制 `.tres` 混入 | 破坏 git diff | 禁止保存 `.tres` 作为设计数据；仅用于引擎自身场景 |
| 跨 EDA 互操作缺失 | 无法与现有团队对接 | M4 规划进出 KiCad 的转换器；现在先保证自有格式稳定 |
| 精度单位混乱 | 浮点误差累积、坐标错位 | 架构层强制 nm `int64` 边界；API 设计时评审；测试覆盖单位换算 |

## 14. ADR 记录机制

重大架构决策以 ADR（Architecture Decision Record）形式归档：

- 位置：`docs/architecture/ADR-<编号>-<主题>.md`
- 模板：见 [文档规范 §5](../conventions/文档规范.md#5-结构模板)
- 本文件的关键决策（nm 坐标、headless、SSOT、JSON-RPC）在 M1 结束前反写为独立 ADR。
