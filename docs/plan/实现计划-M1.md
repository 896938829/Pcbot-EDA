## 1. M1 范围声明

**范围内**：
- 元件库系统（符号 + 元件条目 + SQLite 索引）
- 原理图编辑（数据模型 + 文件格式 + 最小 UI 渲染）
- CLI 骨架（headless 启动 + JSON-RPC + 首批命令）
- AI 中间文件（`.pcbot/last-run.json` + `.pcbot/diagnostics.jsonl` + `.pcbot/commit-msg`）
- GUT 测试骨架 + 首批核心用例
- Skills YAML 首批（覆盖 M1 所有 CLI 命令）

**范围外**（留给 M2+）：
- PCB 布局 / 布线
- DRC / ERC 引擎
- 3D 预览
- 跨 EDA 互操作
- CI 覆盖率门禁、PR 模板、自动化发布

## 2. 验收标准

M1 完成需同时满足：

1. **端到端 smoke 成功**：AI Agent 仅通过 CLI 能完成一个 **LED 闪烁电路**的原理图设计（NE555 或 MCU 最小系统 → LED → 限流电阻 → 电源网）。产出：
   - 可用的 `demo/led_blink.pcbproj`
   - 对应 `demo/led_blink.sch.json`
   - 每步 CLI 调用后 `demo/.pcbot/last-run.json` 被覆盖为本次调用的结构化结果
   - 故意触发一条规则违例后，`demo/.pcbot/diagnostics.jsonl` 追加一条记录；消除违例后再追加一条带 `resolved_by_commit`
2. **AI 可读性**：AI 调用完 CLI 只需读 `.pcbot/last-run.json` 即可判断成败；读 `.pcbot/diagnostics.jsonl` 即可列出未解决违例。无需遍历源文件猜状态。
3. **GUT 单测通过**：核心数据模型（符号、网络、命令路由）有用例；跑 `godot --headless -s addons/gut/gut_cmdln.gd` 全绿。
4. **GUI 可打开**：人类双击 Godot 打开 `demo/led_blink.pcbproj` 能看到原理图（只读渲染也算通过）。
5. **文档对齐**：本计划、架构设计、代码规范、Skills YAML 与实现一致；偏差已以 ADR 记录。

## 3. 工作分解（WBS）

按优先级与依赖排列。规模估算：`S` = 数小时 · `M` = 1–3 天 · `L` = 3 天+（均以专注工时计，不含等待）。

### P1 — 仓库脚手架（S）

- 新建目录：`cli/`、`tests/`、`addons/`、`docs/skills/`。
- 引入 GUT：`addons/gut/`（子模块或手动拷贝，二选一，首次实施时定）。
- 扩展 `.editorconfig`：显式 GDScript 用 Tab；JSON/YAML/Markdown 用 2 空格。
- 扩展 `.gitattributes`：`*.tres text eol=lf`、`*.gd text eol=lf`、SVG 明示 text。
- 加入最小 `project.godot` 自动载入：`EventBus`、`Logger`。
- 验证：`godot --headless --path . --check-only` 无报错。

### P2 — 核心数据模型（M）

创建顺序（下一项依赖上一项的稳定）：

1. `Runtime/core/unit_system.gd` — nm ↔ mm/mil 换算、`Vec2Nm` 类型别名。
2. `Runtime/core/error.gd` — 错误码枚举、结构化错误 `Result` 类。
3. `Runtime/core/event_bus.gd` — 信号注册表。
4. `Runtime/core/logger.gd` — 结构化日志写 stderr。
5. `Runtime/modules/symbol/component_symbol.gd`（`Resource`）。
6. `Runtime/modules/library/component.gd`（`Resource`）。
7. `Runtime/modules/schematic/schematic.gd` / `net.gd` / `placement.gd`（均 `Resource`）。
8. `Runtime/modules/project/design_project.gd`（`Resource`）。

每项配套 GUT 用例：构造、关键字段校验。

### P3 — 文件 I/O（M，可与 P4 并行）

- `Runtime/io/json_stable.gd` — 稳定 JSON 序列化（键字典序、LF、末尾换行、2 空格）。
- `Runtime/io/yaml.gd` — YAML 读写（选 `godot-yaml` 插件或自写最小实现）。
- `Runtime/io/jsonl.gd` — 追加写、按行流读。
- `Runtime/io/svg.gd` — M1 仅需"读 viewBox / 写最小 SVG"，复杂 SVG 延后。
- `Runtime/io/project_fs.gd` — 工程目录抽象：按扩展名派发读写。
- Round-trip 用例：每种格式 "写→读→写"，两次写出字节完全一致。

### P4 — 运行报告与诊断（S，可与 P3 并行）

- `Runtime/io/run_report.gd` — 写 `.pcbot/last-run.json`（覆盖）与 `.pcbot/commit-msg`（覆盖）。
- `Runtime/io/diagnostics_log.gd` — 追加 `.pcbot/diagnostics.jsonl`；提供 `mark_resolved(rule_id, ref, commit_sha)` 追加 `resolved_by_commit` 记录。
- `cli/main.gd` 每次调用末尾统一调用 `RunReport.write(...)`，无论成功失败。
- 用例：
  - CLI 调用失败时 `last-run.json.exit_code` 与进程退出码一致，`errors[]` 非空。
  - `check.basic` 触发规则违例时 `diagnostics.jsonl` 新增一行；再次运行相同 check 不重复写相同 `{rule_id, ref}`（幂等）。
  - `touched_files` 列出本次调用写入的源文件绝对/相对路径集合。

### P5 — CLI 骨架（M）

- `cli/main.gd` — headless 入口；读 stdin 一行 JSON、调路由、写 stdout、按错误类型退出。
- `cli/command_registry.gd` — 命名空间 + 方法映射；支持从模块注册回调。
- `cli/jsonrpc.gd` — JSON-RPC 2.0 请求/响应/错误对象构造与校验。
- 首批命令（依赖 P2–P4）：
  - `project.new` / `project.open` / `project.info`
  - `library.list` / `library.add_symbol` / `library.add_component` / `library.search`
  - `symbol.create` / `symbol.edit_pin` / `symbol.export_svg`
  - `schematic.new` / `schematic.add_page` / `schematic.place_component` / `schematic.connect` / `schematic.annotate`
  - `skills.list` / `skills.get`
  - `check.basic`
  - `run.last`（读 `.pcbot/last-run.json` 回显）
- 用例：每个命令 1 个 smoke（给最小合法参数返回成功）。

### P6 — 元件库（M，依赖 P2/P3/P5）

- `Runtime/modules/library/library_index.gd` — SQLite 封装（`Runtime/io/sqlite.gd` 支撑）。
- 从 JSON 源重建索引：启动时扫 `library/**/*.sym.json`、`library/**/component.json`，写入 `.pcbotlib.sqlite`。
- LRU 内存缓存：按最近访问窗口保留符号资源。
- 命令 `library.search` 支持 `part_number` / `manufacturer` 关键字。

### P7 — 原理图最小 UI（M，依赖 P2/P3）

- `Scenes/Main.tscn` + `Runtime/ui/main_window.gd`：提供文件打开对话。
- `Runtime/ui/schematic_view.gd`：`Control` 节点，绘制符号（SVG → `TextureRect` 或 `_draw()` 矢量）、网络连线。
- 视口：缩放（滚轮）/ 平移（中键拖拽）/ 网格（`10 mm / 1 mm`）。
- **只读**：M1 不做 UI 编辑；编辑通路全靠 CLI。

### P8 — Skills YAML 首批（S，依赖 P5 命名确定）

- 目录：`docs/skills/<namespace>/<method>.yaml`，每个 P5 命令一份。
- 格式：见 [文档规范 §8](../conventions/文档规范.md#8-skills-yaml-格式)。
- 每份至少含：`description`、`triggers`（使用情境）、1 个 `example`、2 个 `common_errors`。

### P9 — 端到端 smoke test（M，最后执行）

- `tests/integration/led_blink_e2e.gd`：
  1. 在临时目录 `project.new`。
  2. `library.add_symbol`（NE555 / LED / 电阻 / VCC / GND）。
  3. `schematic.place_component` × N、`schematic.connect`。
  4. 故意留一个悬空网络跑 `check.basic`，断言 `.pcbot/diagnostics.jsonl` 有对应违例 + `exit_code == 3`。
  5. 补上缺失连接后再跑 `check.basic`，断言 `diagnostics.jsonl` 追加了 `resolved_by_commit` 记录 + `exit_code == 0`。
  6. 用 `git diff` 断言源文件稳定序列化（无非预期差异）。
- 失败立即报错，标示失败步骤 + `.pcbot/last-run.json` 路径。

## 6. 风险清单

| 风险 | 概率 | 影响 | 缓解动作 |
| --- | --- | --- | --- |
| Godot `Resource` 内置 `.tres` 序列化混入设计数据 | 中 | 破坏 git diff | 禁止 `ResourceSaver.save` 到设计目录，统一走 `io/` |
| headless 启动 > 3 s | 中 | 体验差 | 模块懒加载；测算后若仍超，启 daemon 备选（M2 考虑） |
| SVG 解析歧义 | 中 | 符号错位 | M1 仅支持严格 viewBox / 基础图元；更复杂交给专用工具 |
| AI 用例覆盖少 | 高 | 误以为骨架够用 | P9 端到端必须覆盖 LED 闪烁全流程 + 一次违例 + 一次消除 |
| `diagnostics.jsonl` 重复记录 | 中 | 噪声、git diff 膨胀 | 写前按 `{rule_id, ref}` 去重；仅变化时追加 |
| 中间文件向后兼容 | 中 | 升级破坏旧工程 | `last-run.json` / `diagnostics.jsonl` 顶层加 `schema_version`；迁移器在 M2 加 |

## 7. Definition of Done（M1）

- [ ] P1–P9 全部合并入主干。
- [ ] `godot --headless -s addons/gut/gut_cmdln.gd` 退出码 0。
- [ ] `demo/led_blink.pcbproj` 及配套文件存在并通过 P9 集成用例。
- [ ] CLI `skills.list` 输出覆盖所有已实现命令。
- [ ] 每次 CLI 调用都产出 `.pcbot/last-run.json`；`run.last` 命令可回显。
- [ ] `.pcbot/` 在 `.gitignore` 中忽略 `last-run.json` 与 `commit-msg`，保留 `diagnostics.jsonl`。
- [ ] 无 `*.ops.jsonl` / `*.snap.yaml` 遗留引用或代码路径。
- [ ] 本计划文件标注 `状态: stable`，架构决策中的待定项清零或转 ADR。
- [ ] 仓库 `git diff --check` 无 CRLF；`git log` 提交信息符合代码规范。
- [ ] 人类审核者用 Godot 打开 `demo/led_blink.pcbproj` 能看到原理图。

## 8. M2 展望（不在本计划细化）

- PCB 布局数据模型与文件格式（`*.pcb.json`）。
- 基础 DRC：最小间距、未连接引脚、重叠焊盘。
- 布局编辑 UI（仍以 CLI 为主，GUI 只读+微调）。
- 3D 占位预览（矩形拉伸块）。
- CI：GitHub Actions 跑 GUT + 格式化检查；此时起加覆盖率门禁。
