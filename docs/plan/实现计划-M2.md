## 1. 背景

M1 系列收敛后，原理图通道已闭环：库 → 符号 → 原理图编辑 → CLI → GUI 一等编辑面 → Undo/Redo，LED 闪烁端到端跑通（[实现计划-M1](实现计划-M1.md) §2 验收 1）。

按 [工程文档](../工程文档.md) §5 路线图，M2 把"原理图网络"映射到"物理 PCB 布局"，覆盖 **封装数据模型 + PCB 数据模型 + 手动/AI 走线 + 基础 DRC**。**自动布线归 M3，3D 与跨 EDA 互操作归 M4**。

走线策略：M2 不做 auto-route，但提供 CLI `pcb.add_trace / add_via / remove_trace`，让 AI Agent（Claude / Codex 等）外部决策"怎么走"，CLI 仅落数据；人类 GUI 等价手动走线。这是 [ADR-0005](../architecture/ADR-0005-gui-as-first-class-edit-surface.md) "GUI 与 CLI 平权"的延伸。

元件库格式决策：M2 否决 SQLite 切换，继续 JSON 文本（git diff 友好 + 不增项目复杂度），见本期新增 ADR-0008；[ADR-0001](../architecture/ADR-0001-library-index-json-scan.md) SQLite 切换分支 superseded。

## 2. 范围

### 2.1 范围内

1. **封装数据模型** — `ComponentFootprint` (`*.fp.json`)：pads / outline / courtyard / 3d_model_ref?（仅字段占位）
2. **PCB 数据模型** — `Pcb` (`*.pcb.json`)：板形 / 层叠 / 放置实例 / traces / vias
3. **CLI 命令族**：
   - `footprint.*` — `new` / `add_pad` / `set_outline` / `list` / `get`
   - `pcb.*` — `new` / `set_stackup` / `place_footprint` / `move_footprint` / `rotate_footprint` / `remove_footprint` / `add_trace` / `remove_trace` / `add_via` / `remove_via` / `list`
   - `drc.basic` — 间距 / 焊盘重叠 / courtyard 冲突 / 未连焊盘 / 走线短路 / 走线断网
4. **GUI PCB 视口**（一等编辑面）：层切换、放置拖拽、旋转、属性编辑、手动走线工具、DRC 违例高亮
5. **元件库 JSON 性能加固** — 内存索引 + mtime 增量扫描，1500 条 < 200ms
6. **Skills YAML 覆盖** — 所有新增 CLI 命令配套
7. **端到端验收** — 扩展 led_blink demo，AI 通过 CLI 完成 5 元件 PCB 布局 + 手动走线 + DRC 通过

### 2.2 范围外（→ M3 / M4）

- 自动布线 / 智能 push-and-shove（M3）
- ERC / 网络图论 / 阻抗计算（M3）
- 3D 预览（含占位盒子）（M4）
- Gerber / ODB++ 制造输出（M4 或独立里程碑）
- 跨 EDA（KiCad / 其他）互操作（M4）
- DRC R-Tree 加速（M3 性能优化期）

### 2.3 用户已拍板（2026-04-23）

| 项 | 决策 |
| --- | --- |
| 3D 占位盒子 | 推 M4，M2 不做 |
| M1.1 残债（fmt baseline + lambda 重构） | M2 启动前清完（P0） |
| 走线 | M2 含 CLI + GUI 手动走线，不做 auto-route |
| 元件库格式 | 继续 JSON，否决 SQLite 切换 |

## 3. 验收标准

M2 完成需同时满足：

1. **端到端 PCB smoke**：AI 仅 CLI 把 led_blink 5 元件放入 50×50mm 板 + 走完所有信号网络（顶层）+ `drc.basic` 报 0 违例；故意挪一处触发 1 违例 → `.pcbot/diagnostics.jsonl` 追加；修复后追加带 `resolved_by_commit` 行。
2. **GUI 等价编辑**：人类双击工程切到 PCB 视口、拖拽元件、旋转、手动走线、看 DRC 违例红框；任一编辑动作经 `Runtime/modules/pcb/commands.gd` 注册的 CLI 命令落盘（满足 ADR-0005）。
3. **AI 走线工作流**：AI Agent 通过 `schematic.list_nets` + `pcb.list` 读出未走网络，逐条 `pcb.add_trace`，最后 `drc.basic` 校验。Skills YAML 给出标准 prompt 模板。
4. **数据稳序**：`*.pcb.json` / `*.fp.json` 走 `Runtime/io/json_stable.gd`，round-trip 字节一致。
5. **坐标全 nm**：跨模块 API 无 `float`；几何运算（重叠 / 距离）用 `int64`，仅 UI 层转浮点。
6. **DRC 性能基线**：500 焊盘 + 1k 网络 + 200 走线段下 `drc.basic` < 800ms。
7. **元件库性能基线**：1500 条 JSON 库 `library.search` < 200ms（内存索引 + 缓存）；`library.*` 命令行为不变。
8. **Skills 完备**：新增 ~16 个 YAML，`skills.list` 输出齐全。
9. **测试双绿**：GUT 全量 + `tests/lightweight_runner.gd` 双通道。
10. **文档对齐**：本计划、`docs/specs/` 三份格式规范、ADR-0006/0007/0008 与实现一致。

## 4. 工作分解（WBS）

规模：`S` = 数小时 · `M` = 1–3 天 · `L` = 3 天+

### P0 — M1.1 残债清算（M）【M2 启动前必清】

- gdformat 全仓 baseline：先把 `tests/` 中多行 `func():` lambda 抽 helper 或单行化（绕 Godot 4.6 解析 bug），跑 `tools/fmt.sh` 全仓
- 验证：`tools/fmt_check.sh` 绿；`tests/lightweight_runner.gd` 全绿；GUT 全量绿
- 落 [ADR-0006](../architecture/ADR-0006-fmt-baseline.md)：fmt baseline 提交策略 + lambda 写法约束写入 [代码规范](../conventions/代码规范.md)
- 落 [ADR-0008](../architecture/ADR-0008-library-stay-json.md)：固化"元件库继续 JSON、否决 SQLite"决策（superseded ADR-0001 SQLite 切换分支）；说明性能边界与未来重启评估触发条件
- 从 [资源清单](../资源清单.md) 移除 SQLite GDExtension 启用项，标"已否决，参见 ADR-0008"

### P1 — 元件库 JSON 性能加固（S）

- `Runtime/modules/library/index.gd`：内存索引（按 `manufacturer` / `part_no` / `category` 多键 hash）+ 文件 mtime 增量扫描
- 缓存方案在 ADR-0008 决策：方案 A 启动时全扫（1500 条预估 <100ms 可接受）；方案 B `.pcbot/library_cache.bin`（需扩中间文件白名单）。**默认 A**
- `library.search` / `library.list` 性能基线测试入 `tests/perf/library_baseline.gd`
- 不动现有命令签名，回归 M1 库相关测试

### P2 — 封装数据模型 + I/O（M）

- `Runtime/modules/footprint/component_footprint.gd`（`Resource`）
- `Runtime/modules/footprint/pad.gd`（pad_type / shape / size / drill / layers）
- `Runtime/modules/footprint/outline.gd`（轮廓 + courtyard）
- `Runtime/io/footprint_io.gd`（基于 `json_stable.gd`）
- `Runtime/modules/footprint/commands.gd`：`footprint.new` / `add_pad` / `set_outline` / `list` / `get`
- 单测 + round-trip：每命令正/反例
- Skills YAML：5 个

### P3 — PCB 数据模型 + I/O（M）

- `Runtime/modules/pcb/pcb.gd`（`Resource`：板外形 / 层叠 / placements / traces / vias / format_version）
- `Runtime/modules/pcb/stackup.gd`（层定义 + 厚度 + 介电常数占位）
- `Runtime/modules/pcb/footprint_placement.gd`（int64 nm 位姿 + 旋转 + 镜像）
- `Runtime/modules/pcb/trace.gd`（layer + 端点列表 + width + net_id）
- `Runtime/modules/pcb/via.gd`（位置 + 钻孔 + 起止层 + net_id）
- `Runtime/io/pcb_io.gd`
- 单测 + round-trip
- 命令延后 P4 一并

### P4 — PCB CLI 命令 + 走线命令（L）

- `Runtime/modules/pcb/commands.gd` 全套：
  - **布局**：`pcb.new` / `set_stackup` / `place_footprint` / `move_footprint` / `rotate_footprint` / `remove_footprint` / `list`
  - **走线**：`pcb.add_trace` / `remove_trace` / `add_via` / `remove_via`
- 每命令 `Result.data` 含 inverse 所需快照（同 M1.2 模式，参 `Runtime/ui/undo_stack.gd`）
- 走线命令需校验：起止点必须落在同 `net_id` 的焊盘 / via / 既有 trace 端点上
- 错误码新增：`PCB_*` / `FP_*`（统一加在 `Runtime/core/error.gd`）
- 单测：每命令正/反例 + net 一致性 4 种基本拓扑
- Skills YAML：12 个（含 prompt 示例引导 AI 走线工作流）

### P5 — 基础 DRC（M）

- `Runtime/modules/drc/rules/`：
  - `spacing.gd` — 焊盘 / 走线 / via 最小间距
  - `pad_overlap.gd` — 焊盘重叠
  - `courtyard.gd` — 元件 courtyard 冲突
  - `unconnected_pad.gd` — 未走线焊盘（按 net 关联）
  - `trace_short.gd` — 不同 net 走线相交
  - `trace_open.gd` — net 内走线未闭合
- `Runtime/modules/drc/runner.gd`（rule_id 注册 + 顺序扫描；R-Tree 优化推 M3）
- `Runtime/modules/drc/commands.gd`：`drc.basic`
- 集成 `Runtime/io/diagnostics_log.gd`：违例去重、`resolved_by_commit` 标记
- 错误码新增：`DRC_*`
- 单测：每条规则正/反例
- Skills YAML：1 个

### P6 — PCB GUI 视口（一等编辑面）（L）

- `Runtime/ui/pcb_view.gd`（仿 `Runtime/ui/schematic_view.gd`，复用 4-dock + `Runtime/ui/undo_stack.gd`）
- 功能：层切换 toggle、网格、放置 hit-test、旋转 hotkey、属性面板复用
- **手动走线工具**：点击焊盘起始 → 直角折线预览 → 点击落盘 = 调用 `pcb.add_trace`
- **添加 via 工具**：在走线中段点击 = `pcb.add_via`
- DRC 违例红框 + 点击聚焦
- 编辑动作经 `pcb.commands` 落盘 → `EventBus.pcb_disk_changed(path)` 广播 → `pcb_view` 自行 reload
- Undo/Redo：`{forward, inverse}` 命令对入栈
- 手动验收清单：[实现计划-M2-手验清单.md](实现计划-M2-手验清单.md)（P6 末尾产出）

### P7 — 端到端 + Demo 扩展（M）

- 扩展 `tests/integration/led_blink_e2e_test.gd`：原理图 → 选封装 → 放板 → 手动走线 → DRC pass
- demo 添加 `demo/led_blink.pcb.json` + 5 个种子封装：
  - `Assets/footprints/seed/0805.fp.json`
  - `Assets/footprints/seed/SOIC-8.fp.json`
  - `Assets/footprints/seed/DIP-8.fp.json`
  - `Assets/footprints/seed/pin-header-2.fp.json`
  - `Assets/footprints/seed/LED-3528.fp.json`
- `.pcbot/last-run.json` / `diagnostics.jsonl` 路径校验
- AI 走线 prompt 示例落 `docs/skills/pcb/add_trace.yaml`

### P8 — 文档与 ADR（S）

- `docs/specs/2026-04-23-pcb-layout-format.md`（pcb / fp / trace / via 格式权威定义）
- `docs/specs/2026-04-23-drc-rules.md`（首批 6 条规则定义）
- `docs/specs/2026-04-23-ai-routing-workflow.md`（AI 走线标准工作流）
- [ADR-0007](../architecture/ADR-0007-pcb-data-structure.md)：PCB 层叠 + trace/via 数据结构选择
- 复核 ADR-0006 / ADR-0008（已在 P0 落地）
- 更新 [工程文档](../工程文档.md) §5 状态、[资源清单](../资源清单.md)、[CLAUDE.md](../../CLAUDE.md) M2 章节

## 6. 架构铁律自检

- ✅ 所有编辑命令注册 `Runtime/modules/<domain>/commands.gd`（ADR-0005）
- ✅ 模块只依赖 `core/` + 本模块；DRC 读 PCB 数据走显式注入，不跨模块 `preload`
- ✅ `Runtime/io/` 不引领域模型，footprint_io / pcb_io 上层映射
- ✅ 坐标全 `int64` nm；UI 层才浮点
- ✅ CLI 边界统一 `Result`；新错误码加 `Runtime/core/error.gd`（前缀 `PCB_*` / `DRC_*` / `FP_*`）
- ✅ 域事件经 `EventBus.emit_domain` 路由（`footprint.added` / `pcb.placement.moved` / `pcb.trace.added` / `rule.violated`）
- ✅ 中间文件白名单不变（仅 `last-run.json` / `diagnostics.jsonl` / `commit-msg`）
- ✅ 持久化文件带 `format_version`；破坏性升级附 `Runtime/io/migrations/v<N>_to_v<N+1>.gd`

## 7. 风险

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 手动走线 hit-test + 直角折线交互复杂 | P6 工期吹 | 复用 `schematic_view` 矩阵变换；直角折线第一版只支持 90° 双段 L 形，斜线 / 圆角推 M3 |
| 走线 net 一致性校验逻辑漏 | DRC `trace_short` 误报 | P4 命令侧严格校验 `net_id`；P5 DRC 双重校验；单测覆盖 net 拓扑 4 种基本拓扑 |
| `*.fp.json` 字段与 KiCad 互操作冲突 | M4 转换器返工 | 字段命名贴合 KiCad pad/footprint 语义（`pad_type` / `shape` / `drill`）；ADR-0007 立约 |
| 元件库 JSON 扫描在 ≥5000 条时性能退化 | M3+ 用户库膨胀 | P1 内存索引 + mtime 增量扫描；超阈值再走 ADR-0008 重启评估（不在 M2 范围） |
| AI 走线 prompt 不收敛（来回试错） | demo 跑不动 | P7 写 AI 走线工作流文档 + Claude 实测 prompt 模板入 Skills YAML |
| DRC 朴素 O(n²) 性能不达 800ms 基线 | 验收标准 6 失败 | 优化几何过滤（按层 / 按包围盒预筛）；仍不达再考虑插 R-Tree GDExtension（独立 ADR） |

## 8. 估算与排期

- **单人专注：6–8 周**（P0 ~3天；P1 ~2天；P2+P3 ~1.5周；P4 ~2周；P5 ~1周；P6 ~2周；P7+P8 ~1周；缓冲 1周）
- **三人并行：3.5 周**（P0 串行 → P1/P2/P3 并行 → P4/P5 并行 → P6 单人 → P7/P8 收尾）
- **推荐顺序**：P0（残债清算）→ P1（库性能加固）→ P2（footprint）→ P3（PCB 数据）→ P4（PCB CLI）→ P5（DRC）→ P6（GUI）→ P7（E2E）→ P8（文档）

## 9. 验证手段

```bash
# 单测双轨
tools/run_tests.sh

# CLI smoke（布局）
godot --headless -s cli/main.gd -- '{"jsonrpc":"2.0","id":1,"method":"pcb.new","params":{"path":"demo/led_blink.pcb.json","width_mm":50,"height_mm":50}}'

# CLI smoke（走线）
godot --headless -s cli/main.gd -- '{"jsonrpc":"2.0","id":2,"method":"pcb.add_trace","params":{"path":"demo/led_blink.pcb.json","layer":"top","net":"VCC","points_nm":[[0,0],[10000000,0]],"width_nm":250000}}'

# CLI smoke（DRC）
godot --headless -s cli/main.gd -- '{"jsonrpc":"2.0","id":3,"method":"drc.basic","params":{"pcb_path":"demo/led_blink.pcb.json"}}'

# E2E（PCB 通道）
godot --headless --path . -s res://addons/gut/gut_cmdln.gd -gselect=tests/integration/led_blink_e2e_test.gd -gexit

# 性能基线
godot --headless -s tests/perf/drc_baseline.gd
godot --headless -s tests/perf/library_baseline.gd
```
