## 1. 背景与定位修正

M1 / M1.1 当前 GUI 仅有"新建 / 打开 / 载 demo"3 个按钮 + 文字 Info / Status，且
[Runtime/ui/schematic_view.gd:109](../../Runtime/ui/schematic_view.gd#L109) 读 `component_id`
而 [Runtime/modules/schematic/placement.gd:8](../../Runtime/modules/schematic/placement.gd#L8) 写
`component_ref`，导致 demo 永远走"无符号纹理"回退方块——人类打开 demo 看到的是文字提示 + 蓝方块，**完全无法 Review AI 的设计成果**。

**铁律澄清（须修订）**：M1 / M1.1 的"AI 设计、人类审核"被误读为"GUI 仅做被动渲染、不允许编辑"。
正确含义是：AI 与人类**共用同一套领域服务**（CLI 是协议接口），而非禁止人类用 GUI 操作。
人类 Review AI 设计的实际工作流是**在 GUI 里看图、选元件、改 reference、调位置、补连线、查日志**——
没有这些能力，"AI 设计、人类审核"就是空话。

M1.2 目标：把 GUI 升级为完整的"人类 Review + 编辑"工作台，所有变更仍走 CLI 命令落盘
（GUI 是命令的触发器，不引入第二条状态河）。

## 2. 范围声明

**范围内**：

- 修 `component_ref / component_id` 字段错配（P0）
- 架构铁律 §1 表述修订 + ADR-0005（P1）
- 主窗口重构为 4-dock 布局（左：元件库；右：属性；底：日志 + CLI 调试）（P2–P6）
- SchematicView 交互升级（P7–P8）
- 完整人类编辑能力：拖元件、连线、改属性、删除、批量操作（P9–P12）
- 编辑动作 → CLI 命令双向映射（**所有 GUI 编辑必须通过 CLI 命令落盘**，GUI 不直接改文件）
- 撤销 / 重做（基于 CLI 操作栈，不是文件 diff）（P13）
- 文件菜单 / 最近工程 / 保存提示 / 关闭确认（P14）
- 主题 + 用户偏好持久化（P15）
- 测试 + 文档对齐（P16–P17）

**范围外**：

- PCB / DRC / 3D（M2+）
- 协作 / 版本管理 UI（依赖 git，编辑器外）
- 元件库管理面板的"创建符号"工作流（M2 让 AI / CLI 主导，GUI 仅展示）

**与铁律边界**：

- 编辑器**仍然不嵌入 LLM / AI 推理**（无 OpenAI / Anthropic / 本地模型调用）。
- GUI 编辑动作内部调 CLI 命令（同样的 jsonrpc 入口，同样的 Result 反馈，同样写 .pcbot/last-run.json
  与 commit-msg）。
- 所有持久化变更经 `Runtime/modules/<domain>/commands.gd` 注册的方法，**禁止** GUI 直接调
  `JsonStable.write_file` 或绕过 Result。

## 3. 验收标准

M1.2 完成需同时满足：

1. demo 打开后，SchematicView 显示真 SVG 符号（NE555 / R / C / LED）连接完整，**人类一眼可读**。
2. 4 dock 全部就绪：库 / 属性 / 日志 / CLI 调试，可拖动 / 关闭 / 隐藏。
3. 库面板：列出全库 components + symbols；搜索框过滤；选中显示 SVG 预览 + 元数据。
4. 属性面板：点 schematic 上元件 → 显示 reference/value/pos/rot/pins；改 reference 后回车 → 调
   `schematic.set_property` 落盘 + 撤销栈记录。
5. 日志面板：Tab 1 = Logger 输出（颜色按 level）；Tab 2 = `.pcbot/diagnostics.jsonl` 未解决违例列表；
   Tab 3 = `.pcbot/last-run.json` 解析展示。
6. CLI 调试面板：单行 JSON-RPC 输入 → 发送 → 显示响应（折叠 / 高亮）；历史 ≥20 条 + 上下方向键调取。
7. View 交互：mouse hover 高亮 placement 轮廓；点元件高亮并联动属性面板；点引脚 → 高亮该引脚所属
   网络全部连线；状态栏显示 zoom% + 鼠标 mm 坐标 + 选中 ref；按钮 zoom in/out/fit/100% +
   网格 on/off + 网格密度切换。
8. 编辑能力：库面板拖元件到 view 触发 `schematic.place_component`；点元件按住拖动触发
   `schematic.move_placement`（**新 CLI 方法**）；点引脚拖到目标引脚触发 `schematic.connect`（已有）；
   按 Del 触发 `schematic.remove_placement` / `.remove_net`（**新 CLI 方法**）。
9. 撤销 / 重做：Ctrl+Z / Ctrl+Y 调用最近 N 条 CLI 命令的 inverse（操作栈位于 GUI 进程，不入文件）。
10. 文件菜单：File（新建 / 打开 / 关闭 / 最近 / 退出）/ Edit（撤销 / 重做 / 删除）/
    View（4 dock 切换 / 网格 / 主题）/ Help（关于 / 文档）。
11. 关闭工程时若有未保存变更 → 提示对话框（实际 GUI 编辑都即时落盘，提示项仅为 stuck-file 安全网）。
12. 用户偏好持久化：dock 布局 / 主题 / 最近 5 工程 / 网格密度 → `user://prefs.json`。
13. 测试：所有新 CLI 方法（`move_placement / remove_placement / remove_net / set_property`）
    单元覆盖；编辑路径 e2e（拖 → 命令 → 文件 → 重载）integration 覆盖。
14. 文档：架构设计 §1 / §2 修订；新 CLI 方法的 Skills YAML；ADR-0005 立项。

## 7. 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
| --- | --- | --- | --- |
| GUI 编辑动作与文件状态漂移 | 中 | 数据丢失 | 所有编辑必经 CLI 命令落盘；GUI 内存模型每次操作后 reload from disk |
| undo 栈与多用户 / 多会话冲突 | 中 | 复杂度 | M1.2 不支持多窗口；undo 仅本进程；切换工程清栈 |
| 主题切换性能（重绘所有控件） | 低 | UX | 仅启动时切换；不允许运行中 toggle 太频繁 |
| GUI 测试覆盖率低（headless 难跑） | 高 | 回归风险 | UI smoke 走人类手验清单；编辑命令本身覆盖单元 + integration |
| 拖动 / 连线鼠标事件吞没 | 中 | 交互 bug | 严格区分 wiring / drag / pan 模式；状态机转换写测试 |
| dock 布局与 Godot Theme 冲突 | 低 | 显示乱 | 用原生 SplitContainer 不引入 dockable plugin |
